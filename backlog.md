# Backlog

Отобранный набор задач ближайшего развития cal-app: 3 фичи + 4 бага.
Порядок ниже — предлагаемый приоритет (баги 1–2 блокируют фичу «Перенос брони» и друг друга).

## Баги

### BUG-1 — Рабочие часы (08:00–20:00) валидируются только на клиенте
**Приоритет:** высокий — обход через прямой запрос к API создаёт бронь в любое время суток.

**Проблема:** границы рабочего окна захардкожены в
[`apps/frontend/src/components/BookingCalendar.tsx:91-93`](apps/frontend/src/components/BookingCalendar.tsx)
(`slotMinTime="08:00:00"`, `slotMaxTime="20:00:00"`). Сервер (`BookingService.CreateBookingAsync`,
[`apps/backend/src/Cal.Application/Services/BookingService.cs`](apps/backend/src/Cal.Application/Services/BookingService.cs))
эти границы не знает и не проверяет.

**Что сделать:**
- Добавить серверную проверку `startTime`/`endTime` против рабочего окна в `BookingService.CreateBookingAsync`.
- Пока нет настраиваемой доступности (см. фичи), можно захардкодить те же 08:00–20:00 на сервере как константу —
  главное, чтобы прямой вызов API не обходил проверку.

**Acceptance criteria:**
- `POST /bookings` вне рабочего окна → `400` с понятным сообщением об ошибке.
- Существующие happy-path e2e-сценарии (SCN-05) не ломаются.
- Новый regression-тест (backend unit test, см. FEAT-3) на границы окна: ровно 08:00 — ок, 07:59 — 400, 20:00 старт — 400 (если бронь не помещается до конца окна).

### BUG-2 — Нет серверной проверки прошедшего времени при бронировании
**Приоритет:** высокий.

**Проблема:** `CreateBookingAsync` в `BookingService.cs` не сравнивает `request.StartTime` с `DateTime.UtcNow`.
Бронь в прошлое проходит при прямом вызове `POST /bookings`, фронт защищает только косвенно (календарь стартует с «завтра»).

**Что сделать:**
- Добавить проверку `startTime >= DateTime.UtcNow` (с небольшим допуском на сетевую задержку, например `>= UtcNow - 1 min`) в `BookingService.CreateBookingAsync`.

**Acceptance criteria:**
- `POST /bookings` с `startTime` в прошлом → `400`.
- Regression-тест на эту проверку.
- Обе проверки (BUG-1 и BUG-2) стоит делать в одном PR — они в одном методе и логически рядом.

---

### BUG-4 — Гонка при одновременном бронировании одного слота
**Приоритет:** средний-высокий.

**Проблема:** проверка пересечения в `BookingService.CreateBookingAsync` — это `ListAsync` (чтение существующих броней) +
`AddAsync` (запись) без транзакции и без unique constraint в
[`apps/backend/src/Cal.Infrastructure/Persistence/Configurations/BookingConfiguration.cs`](apps/backend/src/Cal.Infrastructure).
Два параллельных `POST /bookings` на один слот теоретически могут оба пройти проверку до того, как любой из них запишется.

**Что сделать:**
- Вариант А (проще): обернуть чтение+запись в транзакцию с уровнем изоляции `Serializable` (или `RepeatableRead`) на уровне `BookingRepository`/`BookingService`.
- Вариант Б (надёжнее, но требует продумать constraint под пересекающиеся интервалы, не просто equality): exclusion constraint в Postgres (`EXCLUDE USING gist`) на диапазон `[StartTime, EndTime)` — потребует расширения `btree_gist` и ручной миграции, EF Core это нативно не поддерживает.
- Для этой итерации достаточно варианта А + regression-тест, вариант Б можно вынести отдельным тикетом, если после нагрузочной проверки останутся дубли.

**Acceptance criteria:**
- Regression-тест: два параллельных `POST /bookings` на один и тот же слот → ровно один `201`, второй `409`.
- Существующий SCN-11 (последовательное бронирование пересекающегося слота → 409) продолжает проходить.

---

### BUG-5 — Нет валидации длины `GuestName`/`GuestEmail`/`Description` на сервере
**Приоритет:** средний.

**Проблема:** DB-ограничения (200/320/1000 символов соответственно) заданы в EF-конфигурациях, но
`BookingService`/`EventTypeService` их не проверяют перед записью — превышение валится в необработанный `500` вместо чистого `400`.

**Что сделать:**
- Добавить явную проверку длины в `BookingService.CreateBookingAsync` (`GuestName` ≤ 200, `GuestEmail` ≤ 320) и в
  `EventTypeService` (`Description` ≤ 1000, заодно свериться с максимумом `Name`, если он тоже ограничен в конфигурации).

**Acceptance criteria:**
- Превышение длины любого из полей → `400` с сообщением, какое поле и какой лимит.
- Regression-тесты на граничные значения (лимит, лимит+1).

## Фичи

### FEAT-1 — Перенос брони (drag-and-drop прямо в календаре)
**Приоритет:** высокий, но **зависит от BUG-1, BUG-2, BUG-4** — реализовывать после них, иначе перенос брони получит те же дыры в валидации, что и создание.

**Идея:** в `BookingCalendar.tsx` (владелец или гость — уточнить на этапе дизайна, скорее всего это фича владельца, т.к. у гостя пока нет способа найти свою бронь повторно) включить `editable`/`eventDrop` у FullCalendar, чтобы существующую бронь можно было перетащить на другой слот прямо в календаре.

**Backend:**
- Новый эндпоинт `PATCH /bookings/{id}` (или `PUT`), принимающий новый `startTime` (и, вероятно, тот же `endTime`, вычисляемый из `durationMinutes` типа события).
- Переиспользовать логику проверки пересечений/рабочих часов/прошедшего времени из `CreateBookingAsync` — стоит вынести в общий приватный метод `ValidateSlotAsync(...)`, которым будут пользоваться и `CreateBookingAsync`, и новый `RescheduleBookingAsync`, чтобы не дублировать BUG-1/2/4 фиксы.
- При проверке пересечений — исключить саму переносимую бронь из выборки существующих (`existingBookings.Where(b => b.Id != bookingId)`), иначе бронь будет конфликтовать сама с собой.

**Frontend:**
- `BookingCalendar.tsx`: `editable={true}`, обработчик `eventDrop` — отправляет `PATCH`, при ошибке (`400`/`409`) — откатывает событие обратно (`dropInfo.revert()`) и показывает сообщение.
- Обновить [`apps/frontend/src/pages/OwnerPage.tsx`](apps/frontend/src/pages/OwnerPage.tsx), если перенос будет доступен из дашборда владельца, а не только из календаря гостя.

**Открытые вопросы для дизайна (уточнить перед реализацией):**
- Кто может переносить бронь — только владелец, или и гость (по прямой ссылке на свою бронь, которой сейчас не существует)?
- Нужно ли уведомление второй стороне о переносе (email) — сейчас уведомлений нет вообще, скорее всего в этой итерации не нужно.

**Acceptance criteria:**
- Перетаскивание брони в календаре на свободный слот → `PATCH /bookings/{id}` → бронь обновлена, календарь отражает новое время без перезагрузки страницы.
- Перетаскивание на занятый слот/вне рабочих часов/в прошлое → откат визуально + сообщение об ошибке, бронь не меняется.
- E2E-тест на happy path переноса + на откат при конфликте.

### FEAT-2 — Отмена брони
**Приоритет:** высокий, независим от FEAT-1 (можно делать параллельно/раньше).

**Backend:**
- Новый эндпоинт `DELETE /bookings/{id}` в `Program.cs` + метод в `IBookingService`/`BookingService` (`CancelBookingAsync` — либо жёсткое удаление строки, либо мягкое удаление через новое поле `Status`, см. ниже).
- Решить: hard delete vs soft delete (добавить `Status` в `Booking`/`BookingEntity` — `Confirmed`/`Cancelled` — потребует миграцию). Soft delete предпочтительнее, если в будущем понадобится история/аудит отмен; hard delete проще и быстрее для этой итерации. Рекомендация: **soft delete**, т.к. `GET /bookings` уже фильтрует по `StartTime >= now`, добавить туда же фильтр `Status != Cancelled` — минимальное изменение, и отменённые брони не теряются безвозвратно.
- `404`, если брони с таким `id` нет; повторная отмена уже отменённой — либо `204` идемпотентно, либо `409`, решить на этапе реализации (рекомендация: идемпотентно `204`).

**Frontend:**
- Кнопка «Отменить» в таблице `OwnerPage.tsx` рядом с каждой броней, с подтверждением (модалка, т.к. действие деструктивное для гостя).
- После отмены — строка убирается из таблицы (или помечается visually, если решили показывать отменённые).

**Acceptance criteria:**
- `DELETE /bookings/{id}` возвращает `204`, повторный вызов — не 500.
- `GET /bookings` больше не включает отменённую бронь.
- Слот освобождается — новая бронь на то же время проходит проверку пересечений (BUG-4 fix должен это учитывать).
- E2E-тест: создать бронь → отменить → убедиться, что слот снова свободен для нового бронирования.

### FEAT-3 — Backend unit tests
**Приоритет:** высокий, **делать в первую очередь** — остальные баги/фичи (особенно BUG-1/2/4 и FEAT-1/2) должны сразу обрастать unit-тестами, а не только e2e; без готового тестового проекта каждый следующий тикет придётся тестировать только через Playwright, что медленнее и менее гранулярно.

**Проблема:** проектов `Cal.*.Tests` нет вообще (`apps/backend/Cal.Web.slnx` ссылается только на 4 основных проекта). Бизнес-логика `BookingService`/`EventTypeService` (пересечения, email-валидация, длительность) покрыта только косвенно через e2e/HTTP.

**Что сделать:**
- Создать проект `apps/backend/tests/Cal.Application.Tests` (xUnit — согласуется с экосистемой .NET/минимальными API, проверить, нет ли уже предпочтения в репо перед выбором фреймворка).
- Добавить его в `Cal.Web.slnx`.
- Замокать/заменить `IBookingRepository`/`IEventTypeRepository` (проверить, есть ли уже интерфейсы репозиториев — если нет, может понадобиться их выделить для тестируемости) — либо использовать EF Core InMemory/Sqlite in-memory provider для тестов, ближе к реальному поведению для проверки уникальных constraint'ов из BUG-4.
- Покрыть тестами: overlap detection (существующая логика + новая после BUG-4), email-валидация, duration validation, плюс все regression-тесты, перечисленные в BUG-1/2/4/5 выше.
- Добавить шаг `dotnet test` в `.github/workflows/ci.yml` (`backend-checks` сейчас делает только `dotnet build`).

**Acceptance criteria:**
- `dotnet test` в CI зелёный, гейтит PR наравне с `dotnet build`.
- Минимум: полное покрытие `BookingService.CreateBookingAsync` (все ветки валидации) и `EventTypeService` создания.
- Каждый последующий тикет из этого backlog (BUG-1/2/4/5, FEAT-1/2) добавляет тесты в этот проект, а не только e2e-сценарии.
