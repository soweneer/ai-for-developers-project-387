# E2E-сценарии

Чек-лист основных пользовательских сценариев приложения. Каждый сценарий имеет
стабильный ID, который используется как префикс в названиях Playwright-тестов
(`e2e/tests/**/*.spec.ts`), чтобы документ и тесты не расходились. Тексты в
кавычках — точные строки из UI (`apps/frontend/src/pages`,
`apps/frontend/src/components`); при обновлении копирайтинга в коде нужно
обновить и этот документ, и связанный тест.

Приложение не имеет авторизации: `/`, `/owner`, `/owner/event-types` и `/guest`
всегда доступны напрямую, выбор роли — чисто навигационный, без контроля
доступа. Это текущее поведение, а не баг, и ниже трактуется как норма.

## Навигация

### SCN-01 — Выбор роли и переключение между ролями
- **Given** пользователь открывает `/`.
- **When** видит заголовок «Вы кто?» с кнопками «Владелец» и «Гость».
- **Then** клик по «Владелец» ведёт на `/owner`, клик по «Гость» — на `/guest`.
- **And** на `/owner` и на `/guest` есть иконка-кнопка с `aria-label="Сменить роль"`,
  которая переключает на противоположную страницу (`/guest` ↔ `/owner`).

## Владелец — типы событий

### SCN-02 — Создание типа события (happy path)
- **Given** владелец на `/owner/event-types`.
- **When** заполняет «Название», «Описание», «Длительность (мин.)» (кратно 15,
  минимум 15) и нажимает «Добавить тип события».
- **Then** появляется бейдж `"{name} · {durationMinutes} мин"` в списке типов,
  форма очищается (длительность возвращается к 30).

### SCN-03 — Клиентская валидация формы типа события
- **When** «Название» пустое → ошибка «Укажите название типа события».
- **When** длительность не кратна 15 → «Длительность должна быть кратна 15 минутам».
- **Then** в обоих случаях запрос `POST /event-types` не отправляется
  (проверяется через перехват сетевых запросов).
- **Примечание:** ошибка «Длительность должна быть больше 0» присутствует в коде
  (`EventTypesPage.tsx`), но недостижима через реальный UI — у `NumberInput`
  задан `min={15}`, и он отклоняет любое значение меньше 15 ещё до обновления
  состояния React (проверено вручную: `.fill('0')` не меняет значение поля).
  Эта ветка не покрыта e2e-тестом по этой причине.

### SCN-03a — Серверная валидация (регрессия на фикс backend)
- **When** `POST /event-types` вызывается напрямую с `name: ""` (в обход клиента).
- **Then** ответ `400` с телом `{ "error": "Name is required." }`.

## Гость — бронирование

### SCN-04 — Пустое состояние гостя
- **Given** в системе нет ни одного типа события.
- **When** пользователь открывает `/guest`.
- **Then** видно «Владелец ещё не создал ни одного типа события», календарь не
  рендерится.

### SCN-05 — Успешное бронирование (happy path)
- **Given** есть хотя бы один тип события, гость на `/guest`.
- **When** кликает по свободному слоту в календаре → открывается модалка
  «Новая встреча» с выбранным (по умолчанию первым) типом события.
- **And** заполняет «Ваше имя» и «Email», нажимает «Назначить встречу».
- **Then** показывается экран подтверждения «Встреча назначена!» с корректным
  временем встречи и кнопкой «Вернуться на главную» (ведёт на `/`).

### SCN-06 — Валидация формы бронирования
- **When** отправка формы без выбранного слота/типа события → «Выберите тип
  события и время встречи».
- **When** отправка формы с пустыми именем/email → «Укажите имя и email».
- **Then** экран подтверждения не показывается.

### SCN-07 — Клик по занятому слоту — no-op
- **Given** в календаре есть занятый (busy) интервал.
- **When** гость кликает на слот, пересекающийся с занятым интервалом.
- **Then** модалка «Новая встреча» не открывается (клик молча игнорируется).

### SCN-08 — Просмотр деталей занятого времени
- **When** гость кликает по уже существующему (серому) событию в календаре.
- **Then** открывается read-only модалка «Занятое время» с названием типа
  события, диапазоном времени, полями «Участник:» и «Email:», без полей ввода.

## Владелец — дашборд

### SCN-09 — Дашборд отражает новую бронь
- **Given** гость только что создал бронь (через API или UI).
- **When** владелец открывает/обновляет `/owner`.
- **Then** в таблице «Назначенные встречи» есть строка с соответствующими
  Гость/Тип события/Время/Email.

### SCN-10 — Пустое состояние дашборда
- **Given** нет ни одной будущей брони.
- **When** владелец открывает `/owner`.
- **Then** видно «Пока нет назначенных встреч», таблица не рендерится.

## Регрессии на фикс backend (пересечение слотов и email)

> Исходно `POST /bookings` не проверял пересечение слотов и не валидировал
> формат email на сервере. Оба бага исправлены (см. `BookingService.cs`);
> сценарии ниже фиксируют корректное поведение и защищают от регрессии.

### SCN-11 — Повторное бронирование пересекающегося слота → 409
- **Given** уже существует бронь на `[start, start+durationMinutes)`.
- **When** `POST /bookings` создаётся ещё раз на пересекающийся интервал (тот же
  или частично overlapping диапазон), с любым другим гостем.
- **Then** ответ `409` с телом `{ "error": "...", "code": "SLOT_UNAVAILABLE" }`,
  вторая бронь не создаётся (проверяется через `GET /bookings`).

### SCN-12 — Бронирование с невалидным email → 400
- **When** `POST /bookings` вызывается с `guestEmail: "not-an-email"`.
- **Then** ответ `400` с телом `{ "error": "Guest email is not a valid email address. (Parameter 'request')" }`,
  бронь не создаётся.
